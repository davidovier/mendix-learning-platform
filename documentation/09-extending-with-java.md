# Extending Mendix with Custom Java

## Overview

Mendix allows developers to extend microflow functionality with custom Java code through Java actions.

"Most application logic can be developed using microflows and nanoflows. If there is a feature missing you can extend Mendix microflows with the use of Java actions."

---

## Creating Java Actions

### Basic Setup

1. Right-click the module and select **Add other** > **Java action**
2. Enter a name for the action
3. Configure parameters and return types through properties pane
4. Save and deploy for Eclipse to generate the Java class

### Parameters and Return Types

Java actions support:
- **Object parameters**: Accepts specific entity types from domain model
- **String/primitive parameters**: Basic data types
- **Return types**: Strings, objects, lists, or other data types

### Code Generation

Studio Pro generates Java code with designated areas for custom code:
- Between `//BEGIN USER CODE` and `//END USER CODE` comments
- Implementation goes in the `executeAction()` method

---

## Implementation in Eclipse

After deploying for Eclipse, developers edit the generated Java class.

### Example: String Manipulation

```java
String productName = this.productParameter.getName(this.getContext());
return new StringBuilder(productName).reverse().toString();
```

---

## Calling Java Actions from Microflows

### Integration Steps

1. Open the microflow where you need the Java action
2. Drag the Java action from **App Explorer** into the microflow
3. Configure input parameters by selecting appropriate variables
4. Use the output variable in subsequent activities

---

## Data Retrieval with XPath

Java actions can retrieve database objects using the Mendix Runtime API.

**Recommendation**: Use "proxy classes generated by Mendix Studio Pro" to avoid raw string XPath queries, ensuring type safety when domain models change.

## Best Practices

### Input Validation

"This simplified approach should be made more robust in a real app by checking and sanitizing the inputs to prevent exceptions or attempts at hacking."

### Using Community Commons

For complex XPath operations, consider the Community Commons module which "provides a powerful way of working with XPaths and performs checks automatically."

### Deployment Workflow

Always use **Deploy for Eclipse** to ensure generated code stays synchronized with action configurations.

## Java Folder Structure (Exam Critical)

When a module contains Java actions, Studio Pro creates a specific folder structure:

```
project/
├── javasource/
│   └── modulename/
│       ├── actions/          # Custom Java action implementations
│       │   └── MyAction.java
│       └── proxies/          # Auto-generated entity proxies
│           └── MyEntity.java
└── userlib/                  # External JAR dependencies
    └── library.jar
```

### Folder Purposes

| Folder | Purpose | Editable? |
|--------|---------|-----------|
| **javasource/[module]/actions/** | Custom Java action code | Yes |
| **javasource/[module]/proxies/** | Generated entity proxy classes | **No** (auto-generated) |
| **userlib/** | External JAR libraries | Yes |

**Critical Warning**: Never manually edit files in the `proxies` folder - they are regenerated automatically when you deploy.

### Proxy Classes

Proxy classes provide type-safe access to domain model entities in Java:
- Generated for each entity in the domain model
- Include getter/setter methods for attributes
- Automatically updated when domain model changes

## Troubleshooting

| Issue | Solution |
|-------|----------|
| **UnsupportedClassVersionError** | Clean deployment directory and match JDK versions |
| **Compile Errors** | Studio Pro identifies syntax errors during deployment |
| **Runtime Errors** | Check Studio Pro console for stack traces |
